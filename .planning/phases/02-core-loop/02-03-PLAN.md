<objective>
Fill in the execution_flow for dan:apply skill -- the most complex of the three core skills. It orchestrates the Execute/Qualify loop: execute a task (in-session by default), spawn a qualifier agent to independently grade the output, route based on status (retry on NEEDS_REVISION, escalate on FAIL with diagnostic routing), and checkpoint progress after each task. Also enhance the executor and qualifier agent prompts with detailed protocols.

Purpose: The apply skill is where work actually happens. The E/Q separation (executor never grades its own work) is DAN's core differentiator. Task checkpointing ensures session interruptions don't lose progress.

Output: One fully-specified skill workflow (dan:apply) and two enhanced agent prompts (dan-executor, dan-qualifier).
</objective>

  <action>
    Replace the placeholder `<execution_flow>` in `.claude/skills/dan-apply/SKILL.md` with a complete workflow. The apply skill orchestrates task execution and qualification.

    The execution_flow must include these steps:

    1. **load_plan**:
       - Accept plan path or phase number as argument
       - If phase number: find the first APPROVED plan in phase directory (parse frontmatter of each PLAN.md)
       - Read plan file: extract objective, tasks, files_modified, requirements
       - Verify plan status is APPROVED (not DRAFT, not already IN_PROGRESS or COMPLETED)
       - If plan is IN_PROGRESS: check STATE.md for last completed task number and resume from there
       - Parse tasks from XML `<task>` elements
       - Read retry counts from STATE.md (persisted from previous session if any): look for `retry_counts` field in STATE.md containing JSON map of task number to count

    2. **start_execution**:
       - Transition plan status from APPROVED to IN_PROGRESS using `dan-tools.cjs frontmatter set <plan-path> status IN_PROGRESS` (validate via lifecycle module)
       - Update STATE.md: "Plan: XX-NN in progress, Task 0 of N"
       - Initialize retry tracking from STATE.md persisted data, or `{}` if fresh start

    3. **execute_and_qualify** (for each task):
       - **Execute** (default: in-session):
         - Read the task's `<action>` instructions
         - Follow instructions to create/modify files per `<files>` list
         - Run any build/test commands as needed
         - On completion, run the task's `<verify>` command
         - Commit task output: `dan-tools.cjs commit "{type}({phase}-{plan}): {task-name}" --files {file-list}`
       - **Qualify** (always via dan-qualifier subagent):
         - Spawn dan-qualifier agent with context:
           - Path to the PLAN.md file
           - Task number being qualified
           - List of file paths modified by the task
           - The task's `<done>` criteria
         - Parse qualifier output using parseQualifierOutput() logic
         - Route based on status:
           - **PASS**: Log success, proceed to next task
           - **PASS_WITH_CONCERNS**: Log concerns in STATE.md decisions section, proceed
           - **NEEDS_REVISION**:
             - Check retry count for this task
             - If retries < 3: read qualifier issues, fix the identified problems, re-commit, re-qualify (loop back to Qualify step)
             - If retries >= 3: run diagnostic_classify step, then escalate to user
             - After each retry increment, persist retry count to STATE.md via `dan-tools.cjs state patch '{"retry_counts": {"taskN": count}}'` so state survives session restart
           - **FAIL**: Run diagnostic_classify step immediately, then escalate to user
       - **Checkpoint** (after each successful task):
         - Update STATE.md: "Plan: XX-NN in progress, Task M of N complete"
         - Persist current retry_counts map to STATE.md via `dan-tools.cjs state patch`

    4. **diagnostic_classify** (triggered by FAIL or exhausted retries):
       - Gather evidence: task spec from plan, qualifier evidence, plan objective
       - Use classifyFailure() logic to determine root cause
       - Route based on classification:
         - **INTENT**: Present to user: "The plan targets the wrong thing. Recommendation: revise plan objective via /dan:plan."
         - **SPEC**: Note spec issue. If possible, revise task description and re-execute from scratch (counts as fresh attempt, resets retry counter). Otherwise escalate.
         - **CODE**: Note implementation bug. Fix and re-qualify (uses normal retry counter).
       - Log classification in STATE.md decisions

    5. **finalize**:
       - After all tasks complete successfully:
         - Transition plan status to COMPLETED using `dan-tools.cjs frontmatter set <plan-path> status COMPLETED`
         - Clear retry_counts from STATE.md (loop complete)
         - Update STATE.md: "Plan: XX-NN complete, ready for unify"
         - Offer next action: "Run /dan:unify to close the loop and produce SUMMARY.md"
       - If execution was interrupted (user chose to stop, or escalation):
         - Leave plan as IN_PROGRESS with checkpoint indicating last completed task
         - Persist retry_counts to STATE.md so next session can resume
         - Log reason in STATE.md

    Each step must have numbered sub-steps with concrete instructions (shell commands, file paths, exact tool invocations). Do not write abstract descriptions -- write executable instructions.

    Also update the "Execution Mode" section to clarify:
    - Default: in-session execution (preserves full context for best implementation quality)
    - Qualification: always via dan-qualifier subagent (independence is the point)
    - The executor and qualifier NEVER share context
